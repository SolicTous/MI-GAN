# Руководство по использованию профилировщика torch.profiler

## Описание
Добавлена возможность включения профилирования обучения через конфигурационный файл. Это помогает определить узкие места в процессе обучения (загрузка данных, forward/backward pass, оптимизация и т.д.).

## Конфигурация
В файле конфигурации эксперимента (например, `configs/experiment/migan_ffhq256.yaml`) добавьте секцию `profile`:

```yaml
profile:
  enabled: false              # Включить профилирование (true/false)
  save_dir: null              # Путь для сохранения результатов (по умолчанию: <log_dir>/profiling)
  record_shapes: true         # Записывать формы тензоров
  profile_memory: true        # Отслеживать использование памяти
  with_stack: true            # Записывать стек вызовов
  num_steps: 10               # Количество шагов для записи
  wait: 1                     # Количество шагов ожидания перед началом записи
  warmup: 2                   # Количество шагов прогрева
  active: 3                   # Количество активных шагов записи
```

## Как использовать

### 1. Включите профилирование
Измените `enabled: false` на `enabled: true` в конфиге:

```bash
# Отредактируйте конфиг
nano configs/experiment/migan_ffhq256.yaml
# Или используйте sed
sed -i 's/enabled: false/enabled: true/' configs/experiment/migan_ffhq256.yaml
```

### 2. Запустите обучение
Запустите обучение как обычно. Профилировщик автоматически:
- Пропустит первые `wait` шагов
- Выполнит `warmup` шагов для прогрева
- Запишет `active` шагов с детальной информацией
- Повторит цикл `num_steps` раз (если настроено)

### 3. Просмотр результатов
Результаты сохраняются в формате TensorBoard Trace:

```bash
# Запустите TensorBoard
tensorboard --logdir log/<experiment_path>/profiling

# Откройте в браузере http://localhost:6006
# Перейдите во вкладку "Profile"
```

## Интерпретация результатов

В TensorBoard Profile вы сможете увидеть:
- **Overview**: Общая статистика использования CPU/GPU
- **Breakdown by operation**: Какие операции занимают больше всего времени
- **Memory usage**: Использование памяти GPU
- **Input pipeline**: Время загрузки данных (поможет выявить проблемы с DataLoader)
- **Kernel statistics**: Детальная информация о CUDA ядрах

## Типичные проблемы и решения

### Проблема: Высокое время в "DataLoader"
**Решение**: Увеличьте `dataset_num_workers_per_gpu` или включите `pin_memory`

### Проблема: Низкая загрузка GPU
**Решение**: Увеличьте размер батча или уменьшите накладные расходы на хосте

### Проблема: Много времени в maintenance
**Решение**: Увеличьте интервалы сохранения чекпоинтов (`snapshot.checkpoint`, `snapshot.image`)

## Отключение профилирования
После диагностики не забудьте отключить профилирование:
```yaml
profile:
  enabled: false
```

Профилирование добавляет накладные расходы и должно использоваться только для отладки.
